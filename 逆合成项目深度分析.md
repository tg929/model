# 逆合成项目深度分析

## 文档目的

这份文档用于作为当前项目后续讨论的主底稿。它不只记录 beam search 修复，而是把当前项目代码的真实结构、only-decoder 的能力边界、训练与评测的主要风险点、以及后续最现实的推进路线统一梳理清楚。

当前状态是：

- only-decoder 的 beam search 第一阶段修复已经落地。
- 新的 beam-fix 测试已经启动。
- 因此现在最重要的事情，不是立刻换模型路线，而是先把当前 only-decoder 的真实上限看清楚。

## 1. 当前项目的真实结构

当前项目不是一个已经集成好的 encoder-decoder 系统，而是三段相对独立的链路：

1. 数据抽取链路
   - `USPTO-full/extract_retrosyn_data.py`
   - 从 atom-mapped `ReactionSmiles` 里抽取真正对 product 贡献原子的 precursor

2. only-decoder 训练/评测链路
   - `USPTO-full/prepare_only_decoder_data.py`
   - `decoder/train_retrosyn_only_decoder.py`
   - `decoder/eval_retrosyn_only_decoder.py`

3. encoder 资产链路
   - `encoder/encoders.py`
   - `encoder/local_bert.py`
   - 已有独立 encoder 权重与 tokenizer
   - 但当前还没有 conditioned encoder-decoder wrapper

也就是说，当前项目“已经有 encoder”和“当前在跑 only-decoder”这两件事，在代码上还没有接起来。

## 2. 当前任务在代码里到底是怎么定义的

### 2.1 数据目标定义是清楚的

`USPTO-full/extract_retrosyn_data.py` 的 `mapped_precursors()` 做的是：

- 只保留单产物反应
- 用 product 端 atom map 找左侧真正参与成键的分子
- reactants 和 reagents 一起扫描
- 只保留 atom-map 与 product 有交集的分子
- 去 atom mapping
- canonicalize SMILES
- precursor 组分按字典序排序后用 `.` 拼接

因此当前任务不是 reaction completion，也不是把所有左侧分子原样抄回去，而是：

- 输入：`product`
- 输出：`true reactants`
- 排除：不贡献原子的溶剂、催化剂、纯试剂

这个定义与单步逆合成目标是一致的。

### 2.2 only-decoder 不是“无条件生成”

`USPTO-full/prepare_only_decoder_data.py` 里，训练样本被写成：

- `source_text = product>>`
- `target_text = reactants`
- `decoder_text = product>>reactants`

因此当前 only-decoder 本质上是一个 prefix-conditioned language model：

`P(reactants | product>>)`

它不是纯 unconditional decoder。它能看到 product，只是看到 product 的方式是“把 product 当成左侧前缀文本”。

### 2.3 训练 loss 的定义是合理的

`decoder/train_retrosyn_only_decoder.py` 里：

- 序列构造为 `[BOS] + source + target + [EOS]`
- `input_ids = seq[:-1]`
- `labels = seq[1:]`
- source 对应位置的 label 被 mask 成 `pad_token_id`

这意味着：

- 模型会利用 `product>>` 作为条件
- loss 只计算 reactants 段
- `product>>` 到第一个 reactant token 的过渡仍然参与训练

这部分逻辑本身没有明显错误。也就是说，当前 only-decoder 至少不是因为“训练目标写错了”而失败。

## 3. 当前 only-decoder 方案的本质优点与结构性限制

### 3.1 它的优点

当前方案最直接的优点有三点：

- 训练路径简单，现成 decoder 就能直接用
- 数据格式统一，只需要一套 decoder tokenizer
- product 到 reactants 的条件建模不需要额外 wrapper

这也是为什么它适合作为第一条 baseline 路线。

### 3.2 它的结构性限制

但从代码结构看，only-decoder 的限制也很明确。

#### 限制一：product 编码和 reactants 生成共用同一套 causal stack

在当前实现里，decoder 既负责“理解 product”，也负责“生成 reactants”。没有单独的 encoder memory，也没有 cross-attention。

这会带来两个问题：

- product 表征完全依赖左侧 token 在 causal self-attention 里被逐层压进隐状态
- 生成 reactants 时，模型访问 product 的方式只是“回看左侧 prefix”，而不是访问一块稳定的编码记忆

对于复杂 product，这种条件建模能力通常弱于显式 encoder-decoder。

#### 限制二：product token 会直接占用上下文预算

当前最大长度过滤是 `256` token，数据里：

- `max_sequence_token_len = 256`
- `p95_sequence_token_len = 157`

因为当前序列是 `product>>reactants` 拼在一起，所以 product 越长，reactants 可生成的余量越小，注意力预算也越被 product 部分占掉。

这和真正的 encoder-decoder 不一样。后者通常是：

- encoder 负责编 product
- decoder 只负责编 reactants 自回归
- product 不直接占 decoder 自回归长度预算

#### 限制三：当前条件信号只有一个文本分隔符 `>>`

现在 product 与 reactants 的边界只是 token `>>`。从工程上它足够简单，但也意味着：

- 模型没有明确的 condition channel
- 没有单独的 encoder hidden states
- 没有任何 bridge 或 retrieval-like condition memory

所以如果任务要求更强的“product 结构理解”，当前路线先天就更吃亏。

## 4. 数据链路的真实特点与风险

### 4.1 这套数据比常见随机切分更难

`USPTO-full/processed_only_decoder/summary.json` 显示：

- `num_pair_rows = 632053`
- `num_unique_products = 606538`
- `train/val/test = 505641 / 63206 / 63206`
- product overlap across splits = `0`

这是 product-exclusive split，不是随机 pair split。

这意味着：

- 测试集 product 在训练集里完全不可见
- 评测更接近“泛化到新 product”
- 分数天然会低于很多更宽松设置下的结果

所以你后面拿结果去对比时，必须始终记住：这里不是最容易的 benchmark 设定。

### 4.2 当前训练实际丢掉了大量频次信息

数据准备时，重复 `(product, reactants)` 对被聚合成一条：

- `num_extracted_rows = 1705815`
- `num_pair_rows = 632053`
- `total_frequency_mass = 1702272`

也就是说，保留下来的 pair 平均频次大约是：

- `1702272 / 632053 = 2.69`

但训练脚本当前只是逐行读取 `train.jsonl`，并没有使用 `count` 做采样加权。

这等价于：

- 数据准备阶段保留了 `count`
- 训练阶段却把它当普通元数据丢在一边

从建模角度，这会削弱高频可靠映射对参数更新的影响。

### 4.3 过滤本身不是主问题

当前过滤掉的 pair 很少：

- `num_pair_rows_before_filter = 633559`
- `num_pair_rows_dropped = 1506`
- drop rate 约 `0.24%`

因此数据过滤本身不是当前性能的主矛盾。

### 4.4 多解 product 的存在是真问题，但不是当前第一主因

同一个 product 可能对应多个 reactants，这是逆合成天然存在的多解性。

但从之前的诊断看，把“同 product 的任一已知 target 都算对”后，top-1 只会有小幅上涨，而不会从十几个点直接跳到表格级别。因此：

- 多解性会影响 strict exact match
- 但不是当前 only-decoder 成绩偏低的唯一核心原因

## 5. 训练链路的真实特点与风险

### 5.1 当前 best checkpoint 是按 teacher-forced loss 选出来的

`decoder/train_retrosyn_only_decoder.py` 里的验证是：

- `val_loss`
- `val_perplexity`

它不是 beam top-k generation validation。

所以当前 `epoch4` 的含义是：

- 在 teacher-forced validation 下，`epoch4` 最优

它不等于：

- 在生成 top-k 指标下，`epoch4` 一定最优

这也是为什么 `epoch4` 虽然仍然有价值，但修完 beam search 后最好还要把 `epoch3/epoch4/epoch5` 做同口径生成对比。

### 5.2 训练 recipe 目前偏基础

当前训练脚本有这些特征：

- AdamW
- 固定学习率
- 没有 warmup
- 没有 scheduler
- 没有按 `count` 加权
- 没有针对生成指标做 early selection

这不代表它不能工作，但意味着：

- 当前 recipe 更像 baseline
- 还远不是“逆合成任务专门打磨过的训练策略”

所以如果 beam-fix 后结果仍不理想，也不能立刻把责任全压到“decoder 架构没戏”上。

### 5.3 当前训练与最终目标之间仍有一层错位

训练在优化：

`token-level next-token likelihood`

你真正汇报时在看的是：

- top-k exact
- top-k canonical
- top-k maxfrag

这两者不是同一个目标。

这会导致一个现实问题：

- 一个 checkpoint 的 `val_loss` 更低
- 但它的 beam top-10 可能反而不如另一个 checkpoint

如果后续你要更系统地推进，这个错位必须尽快补上。

## 6. 评测链路的真实特点与风险

### 6.1 之前的 beam search 问题只污染评测，不污染训练

这一点需要明确写死：

- 之前 beam search 的问题发生在 `decoder/model.py` 的生成逻辑
- 不发生在训练 loss 计算路径上

因此：

- `epoch4` checkpoint 本身仍然有价值
- 之前不干净的是评测，不是训练结果本体

### 6.2 当前评测指标是严格的

`decoder/eval_retrosyn_only_decoder.py` 当前输出：

- top-k exact match
- top-k canonical match
- top-k largest-fragment match
- top-1 invalid SMILES rate

其中 exact 和 canonical 都是严格匹配，不会因为“化学上有别的可行路线”而给分。

这意味着：

- 指标很干净
- 但对逆合成这种天然多解任务来说，也偏保守

### 6.3 长任务评测之前有落盘风险，现在已修复

旧评测的问题是：

- 预测和计数长时间保留在内存
- 直到整个任务结束才一次性写文件

现在已经改成：

- predictions 边跑边追加写 JSONL
- metrics 每 `N` 条样本重写一次
- milestone metrics 可保存 `1000 / 2000 / 3000 ...` 的阶段性结果

这让全测试集评测更适合作为生产化实验流程。

## 7. 从代码角度看，当前最值得重视的几个瓶颈

结合代码结构，我认为当前优先级最高的瓶颈依次是：

### 7.1 第一瓶颈：先拿到干净的 beam-fix 结果

这是当前第一优先级，不需要争论。

原因很简单：

- 旧结果已知被 beam search 污染
- 在新结果出来前，讨论“only-decoder 行不行”仍然容易偏

### 7.2 第二瓶颈：checkpoint 选择标准与最终目标不一致

如果 beam-fix 后 `epoch4` 仍然最好，那很好。

但如果 `epoch3` 或 `epoch5` 的 top-k 更高，你现在按 `val_loss` 选最佳模型的策略就要调整。这个问题是纯工程问题，不是模型哲学问题，优先修复收益很高。

### 7.3 第三瓶颈：训练阶段没用上 `count`

当前数据已经有频次信息，但训练完全没利用。

这是一个投入小、潜在收益不低的改进点。它比“直接大改架构”更适合做第二轮实验。

### 7.4 第四瓶颈：only-decoder 的条件建模上限

如果 beam-fix 后，结果仍明显低于你的目标，那下一层真正要面对的问题，就是：

- 只靠 prefix `product>>`
- 用同一套 causal decoder 去同时理解 product 和生成 reactants

这条路线可能已经开始逼近它的结构性上限。

## 8. encoder 对后续路线意味着什么

### 8.1 你现在的 encoder 不是摆设，它和 decoder 有一个很关键的兼容点

从代码和配置看：

- encoder hidden size = `2048`
- decoder `650M` hidden size = `2048`

也就是说，二者的主通道维度是对齐的。

这件事很重要，因为它意味着：

- 如果后面做 conditioned wrapper
- 不一定非要先做很重的维度映射层
- 至少在张量主维度上，bridge 设计会更顺手

### 8.2 tokenization 也没有明显阻塞

encoder 和 decoder 都采用 SMILES regex tokenizer，且都能处理 `>>` 这类 reaction token。

如果后面做 encoder-decoder：

- encoder 完全可以只吃 product
- decoder 单独吃 reactants 侧
- 不需要强迫两边共用同一个 vocab

这对“冻结 encoder，只训练 decoder 侧条件接入层”很友好。

### 8.3 当前缺的不是 encoder 权重，而是连接方式

现在代码里缺的是：

- cross-attention block
- 或 adapter / bridge
- 或 prefix-memory / soft prompt style condition 注入

不是缺 encoder 本体。

所以后续路线的重点，不是再去折腾 encoder 预训练，而是设计一个尽量小、尽量稳的接法。

## 9. 如果后续转向 encoder-decoder，哪条路线最现实

我认为从当前代码出发，最现实的路线分两档。

### 9.1 路线 A：冻结 encoder，给 decoder 加 cross-attention

这是最标准、最符合任务定义的方案：

- encoder 编码 product
- decoder 自回归生成 reactants
- decoder 在每层或部分层通过 cross-attention 读取 encoder hidden states

优点：

- 任务形式最干净
- product 条件记忆最明确
- 比 prefix-only decoder 更符合 `product -> reactants`

代价：

- 需要改动 decoder block 结构
- 需要新增 conditioned wrapper、训练脚本、推理脚本

### 9.2 路线 B：冻结 encoder，先做一个轻量 bridge / adapter

如果你想先做低风险验证，可以先不上全套 cross-attention，而是：

- encoder 产出 product hidden states
- 用一个轻量模块把 encoder 输出压成少量 condition vectors
- 再把这些 condition vectors 注入 decoder

实现方式可以有几种：

- soft prompt / prefix token 风格
- 只在部分层注入 adaptor
- encoder summary + gated residual 注入

优点：

- 改动更小
- 更快验证 encoder 条件是否真的能抬升

缺点：

- 上限通常不如标准 cross-attention 清晰

### 9.3 对你当前项目，我更倾向的顺序

我更倾向：

1. 先把 only-decoder 干净基线拿出来
2. 再做 generation-aware checkpoint 选择
3. 再做 `count` 加权版本
4. 如果仍离目标明显偏远，再上冻结 encoder 的 conditioned wrapper

因为这样每一步的归因都最清楚。

## 10. 我对当前项目状态的判断

如果只基于当前代码和已经确认的信息，我的判断是：

### 10.1 现在还不能得出“only-decoder 不行”的结论

原因是：

- 旧评测已知被 beam search 污染
- 干净结果还在跑
- 当前 best checkpoint 的选法也和最终 generation 目标不完全一致

因此现在否定 only-decoder，还太早。

### 10.2 但也不能假装 only-decoder 没有结构上限

即使 beam-fix 后分数有明显抬升，当前路线依然有先天限制：

- 没有显式 encoder condition memory
- 没用上 pair 频次
- 没有 generation-aware checkpoint selection

所以它更像一个“必须先跑清楚的 baseline”，而不是最终最优形态。

### 10.3 encoder 是你后续真正能拉开差距的牌

你已经有训练好的 encoder，而且不需要微调它。

这在当前项目里是很强的条件，因为：

- encoder 和 decoder hidden dim 对齐
- 数据任务定义也已经清楚
- 真正缺的只是 wrapper 与训练/推理路径

如果 only-decoder clean baseline 仍不够，下一步最值得投入的，不是继续盲目堆 decoder epoch，而是让 encoder 正式参与条件建模。

## 11. 建议的后续执行顺序

### 第一阶段：把当前 only-decoder 看准

1. 等 beam-fix 的 1000 条结果出来
2. 判断 top-1 / top-10 是否有明显抬升
3. 如果抬升明显，再补全测试集

### 第二阶段：让 checkpoint 选择与目标一致

1. 用同一口径重测 `epoch3 / epoch4 / epoch5`
2. 选 generation top-k 真正最优的 checkpoint
3. 后面训练时补上小规模 generation validation

### 第三阶段：先做低成本训练改进

优先考虑：

- `count` 加权采样
- 或恢复部分重复样本训练信号
- 保持其他设置尽量不变，先看单因素收益

### 第四阶段：如果仍不够，再上 encoder-decoder

建议优先级：

1. 冻结 encoder
2. 先尝试轻量 bridge / adapter
3. 如果收益明确，再决定是否上完整 cross-attention decoder

## 12. 当前结论

当前项目已经具备三样关键东西：

- 定义清楚的 `product -> reactants` 数据链路
- 可运行的 only-decoder baseline
- 可复用的独立 encoder 资产

真正还没完成的，是把这三者按“正确优先级”衔接起来。

对当前阶段，最重要的不是继续猜，而是按这个顺序推进：

- 先拿干净的 only-decoder 结果
- 再修 checkpoint 选择逻辑
- 再补频次信号
- 最后决定是否正式转向 encoder-conditioned 路线
